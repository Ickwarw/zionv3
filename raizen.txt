VISÃO GERAL DA ARQUITETURA RAIZEN
=====================================================================
O ecossistema Raizen é dividido em 3 camadas principais:

1) Camada de IA (raizen.py)
   - Responsável por conversar com provedores de LLM (OpenAI, Google, Anthropic).
   - Controla prompt, histórico, tokens e custo.

2) Camada de dados/aprendizado (raizen_ml_system.py)
   - Responsável por persistir conversas, treinamentos, popups, insights, feedback e logs no MySQL.
   - Permite ciclo de aprendizado contínuo com aprovação humana.

3) Camada de orquestração (raizen_integrated.py)
   - Une IA + banco + regras de negócio.
   - Aplica treinamentos aprovados no contexto, cria insights e alimenta dashboard.

Arquivos auxiliares:
- raizen_setup_config.py: setup e validação de chaves de API.
- raizen_exemplo_uso.py: demonstrações práticas de uso.

1) ARQUIVO: raizen.py (NÚCLEO DE IA)
=====================================================================
Classe principal: RaizenAssistant

1. __init__(config_file='raizen_config.json')
   Finalidade:
   - Inicializar o agente com configuração, métricas e chaves de API.

   Entradas:
   - config_file (str): caminho do JSON com modelos, prompts e parâmetros globais.

   Saída:
   - Não retorna valor; prepara estado interno da instância.

   Fluxo interno:
   - Chama _load_config().
   - Inicializa conversation_history (lista).
   - Inicializa metrics (requests, tokens, custo, contagem por provider/modelo).
   - Lê OPENAI_API_KEY / GOOGLE_API_KEY / ANTHROPIC_API_KEY das variáveis de ambiente.

   Conexões:
   - Sistema de arquivos (config JSON).
   - Variáveis de ambiente.

   Pontos de atenção:
   - Se config estiver ausente/inválido, a inicialização falha.
   - Se API keys estiverem vazias, chamadas externas falharão em runtime.


2. _load_config(config_file)
   Finalidade:
   - Ler e validar parsing do JSON de configuração.

   Entradas:
   - config_file (str).

   Saída:
   - dict com configuração completa.

   Fluxo interno:
   - open(..., encoding='utf-8') + json.load().
   - Trata FileNotFoundError e JSONDecodeError com log + raise.

   Conexões:
   - Arquivo local de configuração.


3. get_available_models(provider=None, include_free=True, include_paid=True)
   Finalidade:
   - Montar catálogo de modelos disponíveis a partir do config.

   Entradas:
   - provider: filtra por openai/google/anthropic (ou todos quando None).
   - include_free/include_paid: controla subconjuntos.

   Saída:
   - list[dict] com metadados de modelos (id, nome, tipo, provider, categoria, preço etc.).

   Fluxo interno:
   - Percorre config['modelos_ia'] por provider.
   - Anexa campos adicionais: provider e tipo ('gratuito'/'pago').

   Conexões:
   - Estrutura interna do config JSON.


4. _prepare_openai_request(model, messages, temperature, max_tokens)
   Finalidade:
   - Converter dados internos para payload compatível com OpenAI Chat Completions.

   Entradas:
   - model, messages, temperature, max_tokens.

   Saída:
   - dict com {url, headers, payload} pronto para requests.post().

   Conexões:
   - API OpenAI: https://api.openai.com/v1/chat/completions


5. _prepare_google_request(model, messages, temperature, max_tokens)
   Finalidade:
   - Adaptar estrutura de mensagens para formato Gemini.

   Entradas:
   - model, messages, temperature, max_tokens.

   Saída:
   - dict com url (inclui key), headers e payload no formato contents/parts.

   Fluxo interno:
   - Converte role interno para role Gemini ('user'/'model').
   - Empacota texto em parts.

   Conexões:
   - API Google Gemini:
     https://generativelanguage.googleapis.com/v1beta/models/{model}:generateContent


6. _prepare_anthropic_request(model, messages, temperature, max_tokens)
   Finalidade:
   - Montar requisição no formato Claude Messages API.

   Entradas:
   - model, messages, temperature, max_tokens.

   Saída:
   - dict com url, headers e payload.

   Fluxo interno:
   - Separa mensagem system das mensagens conversacionais.
   - Mantém roles esperados pela Anthropic.

   Conexões:
   - API Anthropic: https://api.anthropic.com/v1/messages


7. _parse_openai_response(response)
   Finalidade:
   - Normalizar resposta OpenAI para formato interno único.

   Entradas:
   - response (dict) bruto da API.

   Saída:
   - dict: {content, tokens_input, tokens_output, tokens_total}.

   Pontos de atenção:
   - Estrutura inesperada dispara erro (KeyError/IndexError).


8. _parse_google_response(response)
   Finalidade:
   - Normalizar resposta Gemini para formato interno.

   Entradas:
   - response (dict).

   Saída:
   - dict de conteúdo e tokens.


9. _parse_anthropic_response(response)
   Finalidade:
   - Normalizar resposta Claude para formato interno.

   Entradas:
   - response (dict).

   Saída:
   - dict de conteúdo e tokens.


10. _make_api_request(provider, request_data, retry_attempts=3)
    Finalidade:
    - Executar chamada HTTP com estratégia de retry.

    Entradas:
    - provider, request_data, retry_attempts.

    Saída:
    - JSON da resposta da API.

    Fluxo interno:
    - requests.post(..., timeout=config.timeout_segundos).
    - response.raise_for_status().
    - Em falha de rede/HTTP: retry com backoff exponencial 2^tentativa.

    Conexões:
    - Rede externa + requests.

    Pontos de atenção:
    - Se todas tentativas falharem, exceção sobe para camada chamadora.


11. _calculate_cost(model_info, tokens_input, tokens_output)
    Finalidade:
    - Estimar custo monetário local da inferência.

    Entradas:
    - model_info com preco_input/preco_output.
    - tokens_input, tokens_output.

    Saída:
    - float (USD).

    Regra:
    - custo = (tokens_input/1_000_000 * preco_input) + (tokens_output/1_000_000 * preco_output)


12. _update_metrics(provider, model, tokens, cost)
    Finalidade:
    - Atualizar telemetria local da sessão.

    Entradas:
    - provider, model, tokens(normalizados), cost.

    Saída:
    - Sem retorno; altera self.metrics.


13. chat(message, provider, model_id, prompt_type='analise_marketing', temperature=None, max_tokens=None, reset_history=False)
    Finalidade:
    - Método central de conversação do Raizen.

    Entradas:
    - message: texto do usuário.
    - provider/model_id: roteamento para LLM.
    - prompt_type: seleciona system prompt no config.
    - temperature/max_tokens opcionais.
    - reset_history: limpa contexto conversacional antes da nova pergunta.

    Saída:
    - dict completo com resposta, metadados de modelo, tokens, custo, timestamp e parâmetros.

    Fluxo detalhado:
    1) (Opcional) limpa histórico.
    2) Define defaults de temperatura/max_tokens via config.
    3) Se histórico vazio, injeta system prompt.
    4) Adiciona mensagem user ao histórico.
    5) Prepara request específico do provider.
    6) Chama API com retry.
    7) Faz parse da resposta.
    8) Adiciona resposta assistant ao histórico.
    9) Busca metadado do modelo e calcula custo.
    10) Atualiza métricas.
    11) Retorna payload normalizado.

    Conexões:
    - prompts do config.
    - APIs externas de IA.
    - estado de histórico e métricas.

    Pontos de atenção:
    - provider inválido gera ValueError.
    - Histórico cresce indefinidamente se não houver reset/gestão externa.


14. analyze_marketing_data(dados, provider, model_id)
    Finalidade:
    - Wrapper de caso de uso de marketing analytics.

    O que faz:
    - Serializa dados em JSON no prompt e solicita insights, tendências, oportunidades, recomendações, métricas e projeções.
    - Executa chat com prompt_type='analise_marketing' e reset_history=True.


15. analyze_customer_churn(dados_clientes, provider, model_id)
    Finalidade:
    - Wrapper para análise de risco de churn.

    O que faz:
    - Solicita identificação de clientes em risco, fatores causais e plano de retenção.
    - Usa prompt_type='recuperacao_clientes'.


16. create_acquisition_strategy(perfil_target, orcamento, provider, model_id)
    Finalidade:
    - Gerar plano de aquisição com orçamento.

    O que faz:
    - Pede canais, distribuição de verba, KPIs, timeline, CAC e conversão.
    - Usa prompt_type='captacao_clientes'.


17. identify_process_failures(dados_processo, provider, model_id)
    Finalidade:
    - Diagnosticar gargalos operacionais/comerciais.

    O que faz:
    - Solicita falhas, impacto, quick wins, melhorias de médio/longo prazo e ROI.
    - Usa prompt_type='analise_falhas'.


18. create_dashboard_insights(dados, tipo_dashboard, provider, model_id)
    Finalidade:
    - Gerar estrutura orientada a dashboard com insights acionáveis.

    O que faz:
    - Solicita KPIs, visualizações, alertas, tendências e JSON de implementação frontend.
    - Usa prompt_type='dashboard_previsoes'.


19. predict_sales(historico_vendas, periodo_previsao, provider, model_id)
    Finalidade:
    - Criar previsão de vendas com análise de tendência/sazonalidade.

    O que faz:
    - Solicita cenários (otimista, realista, pessimista), confiança e contingência.
    - Usa prompt_type='dashboard_previsoes'.


20. get_metrics()
    Finalidade:
    - Retornar snapshot das métricas acumuladas da instância.

    Saída:
    - timestamp, métricas, tamanho do histórico e custo médio por request.


21. reset_metrics()
    Finalidade:
    - Zerar telemetria sem reiniciar objeto.


22. save_conversation(filename)
    Finalidade:
    - Persistir histórico + métricas no disco para auditoria/replay.

    Conexões:
    - Sistema de arquivos local.


23. load_conversation(filename)
    Finalidade:
    - Reidratar histórico salvo anteriormente.


24. main()
    Finalidade:
    - Exemplo de execução standalone do assistente.

Classe principal: RaizenIntegrated

1. __init__(config_file='raizen_config.json', db_host='localhost', db_user='root', db_password='', db_name='zioncrm')
   Finalidade:
   - Subir stack completo integrada.

   Fluxo:
   - Inicializa RaizenAssistant.
   - Cria RaizenDatabase e conecta.
   - Cria RaizenMLSystem.
   - Define IDs de sessão e flags de aprendizado.

   Pontos de atenção:
   - Falha de conexão MySQL dispara ConnectionError.


2. set_user(user_id, user_name)
   - Encaminha contexto de usuário para camada ML.


3. chat(message, provider, model_id, prompt_type='analise_marketing', temperature=None, max_tokens=None, learn_from_interaction=True)
   Finalidade:
   - Chat com aprendizado contínuo e persistência total.

   Fluxo detalhado:
   1) Cria conversa se não houver atual.
   2) Salva mensagem do usuário no banco.
   3) Busca treinamentos aprovados relevantes (se learning habilitado).
   4) Enriquece mensagem com contexto histórico de aprendizado.
   5) Chama assistant.chat().
   6) Salva resposta do assistente com tokens, custo e latência.
   7) Detecta possíveis insights automáticos por palavras-chave.
   8) Loga atividade operacional.
   9) Opcionalmente salva interação como candidato de treinamento.
   10) Retorna payload normalizado para frontend.

   Saída:
   - dict com success, IDs, resposta, modelo, tokens, custo, tempo, timestamp e uso de treinamento.

   Tratamento de erro:
   - log de erro + log_activity(status='error') + resposta {'success': False, 'error': ...}.


4. _get_model_info(provider, model_id)
   - Busca metadados do modelo para enriquecer criação da conversa.


5. _enhance_message_with_training(message, trainings)
   Finalidade:
   - Injetar contexto curto de aprendizados aprovados para melhorar resposta.

   Regra:
   - Usa até 3 treinamentos com expected_output.


6. _save_as_training_candidate(message, response, category, tokens_used)
   Finalidade:
   - Capturar interação útil para curadoria humana posterior.

   Regra:
   - Só salva se tokens_used > 100 (threshold de relevância).


7. _analyze_response_for_insights(message, response, category)
   Finalidade:
   - Detectar gatilhos semânticos simples (churn/opportunity/problem) e gerar insight.

   Observação:
   - Método heurístico por palavras-chave; pode gerar falso positivo/negativo.


8. get_pending_popups()
   Finalidade:
   - Fornecer lista de popups pronta para consumo em frontend.

   O que agrega:
   - serializa datas para ISO.
   - gera botões de ação por tipo de popup.


9. _generate_popup_buttons(popup)
   Finalidade:
   - Traduz tipo/ação do popup em comandos de UI.

   Regras:
   - approve_training => Aprovar/Rejeitar/Detalhes.
   - view_customer_analysis => Ver/Tomar ação/Dispensar.
   - default => Confirmar/Cancelar ou OK.


10. respond_to_popup(popup_id, action, response=None, data=None)
    Finalidade:
    - Aplicar ação do usuário e atualizar estado de popup.

    Casos importantes:

11. analyze_customer(customer_id, customer_data, analysis_types=None)

    - Pipeline de análise de cliente com IA generativa + persistência estruturada.

    Fluxo:

      a) cria prompt específico,
      b) consulta IA (OpenAI gpt-4o-mini no código atual),

    - Parsing heurístico; sensível a formatos de resposta inesperados.

15. get_dashboard_data()
    Finalidade:

6) O QUE O AGENTE RAIZEN FAZ (DETALHADO)
=====================================================================
- Atua como copiloto de marketing/comercial com LLM multi-provider.
- Traduz dados brutos em análises estratégicas (churn, aquisição, falhas, previsões, dashboards).
- Monitora custo e uso por modelo/provider.
- Em modo integrado, persiste toda trilha de conversa no MySQL.
- Implementa aprendizado contínuo supervisionado:
  - coleta candidatos de treinamento,
  - solicita aprovação humana,
  - reutiliza treinamentos aprovados no contexto futuro.
- Cria popups operacionais para ação humana em pontos críticos.
- Gera insights automáticos e alertas de risco (clientes, popups urgentes, insights críticos).
- Oferece base de auditoria: logs, feedback, estatísticas e configuração dinâmica no banco.

=====================================================================
8) LIMITAÇÕES E PONTOS DE MELHORIA IDENTIFICADOS
=====================================================================
1. Parsing heurístico em _parse_customer_analysis
   - Usa regex simples para score; pode interpretar número errado dependendo do texto.

2. Crescimento de histórico em chat
   - conversation_history pode crescer sem limite se não houver política de truncamento.

3. Aprendizado por palavra-chave em _analyze_response_for_insights
   - Heurística básica; ideal evoluir para classificação mais robusta.

4. Dependência de schema SQL
   - O modo integrado exige tabelas/views/procedures presentes e coerentes.

5. Falta de validação forte de payloads
   - Há serialização e saneamento, mas não há validação semântica rígida por schema em todas as entradas.

9) FLUXO RESUMIDO DE EXECUÇÃO (MODO INTEGRADO)
=====================================================================
Passo 1: Usuário envia mensagem -> RaizenIntegrated.chat()
Passo 2: Sistema cria/retoma conversa no MySQL
Passo 3: Salva mensagem do usuário
Passo 4: Busca treinamentos aprovados e enriquece contexto
Passo 5: Chama provedor de IA via RaizenAssistant.chat()
Passo 6: Salva resposta + tokens + custo + latência
Passo 7: Gera insights automáticos (quando detectado)
Passo 8: Registra log de atividade
Passo 9: Opcionalmente salva interação como candidato de treinamento
Passo 10: Retorna payload formatado para frontend

10) CONCLUSÃO
=====================================================================
O Raizen é um agente de IA comercial/marketing com arquitetura modular e rastreável,
capaz de operar em modo simples (somente APIs de IA) ou modo enterprise (IA + banco +
aprendizado supervisionado + monitoramento operacional). As funções cobrem ciclo completo
de geração de resposta, persistência, curadoria de treinamento, insights e suporte a dashboard.
